## Раздел 1: Критические несоответствия (P0 — нет в спеках)

### 1.1 «Дашборд — это тоже markdown-файл»
**VISION §8 (дословно):**
> "Сам дашборд — это тоже markdown-файл. Не закрытая конфигурация в JSON, а читаемая, версионируемая, синхронизируемая через git заметка особого типа. Я могу открыть её в Vim и понять структуру. Могу скопировать дашборд другу и он откроется у него."

**Реальность**: Dashboard-конфиг живёт в `data.json` → проприетарный JSON, непрозрачный, не открывается в Vim с понятной структурой.

**В спеках**: Ни `DASHBOARD_V2_SPEC.md`, ни `UI_DESIGN_ARCHITECTURE.md`, ни `UI_MODERNIZATION_PLAN.md` **не содержат ни одного упоминания** этого требования. `DASHBOARD_V2_SPEC.md §14` явно перечисляет "что НЕ меняем в V2", но `data.json` туда не попал — значит статус неопределён.

**Последствие**: Это не просто техническая деталь — это философская идентификация продукта. "Всё — твои файлы, всё — навсегда" — если дашборд живёт в JSON, этот принцип нарушается.

**Требуемое действие**: Нужно явное решение: (A) запланировать миграцию `ProjectDefinition` в YAML-frontmatter формат читаемого markdown-файла, или (B) зафиксировать осознанный компромисс с обоснованием — "data.json выбран потому что X, dashboard-as-markdown деферировано до V3". Сейчас нет ни того, ни другого.

### 1.2 «Проактивные подсказки по типам данных»
**VISION §6 (дословно):**
> "Видишь поле 'цена сеанса'? Покажу сумму за месяц, среднее, рост к прошлому периоду."
> "Видишь поле 'оценка боли' по шкале 0-10? Построю динамику, выделю тренд, отмечу аномалии."
> "Видишь связи между клиентом и сеансами? Покажу частоту визитов, среднюю длину паузы."
> "Это **проактивные подсказки**, а не сложная конфигурация."

**В спеках**: Нулевое покрытие. Ни тикета, ни дизайна, ни упоминания в `UI_DESIGN_ARCHITECTURE.md`. В тексте Vision называется «главная инновация» — «пользователь видит данные, плагин предлагает осмысленные агрегаты».

**Последствие**: Без этой системы продукт остаётся "умной таблицей". С ней — он становится аналитическим партнёром. Это разница в позиционировании.

**Требуемое действие**: Новый тикет `#059 — SmartSuggest: проактивные подсказки по типам данных`. Минимальная MVP-версия: при добавлении числового поля в базу → всплывающая suggestion strip "Хотите добавить Stats-виджет с агрегатом по этому полю?". При обнаружении Relation-поля → suggestion "Показать связанные записи в sub-base?"

### 1.3 «Прозрачность контракта: курсор по колонке → превью файла»
**VISION §3 (дословно):**
> "Курсор по колонке — подсветка соответствующей строки в превью заметки."
> "Двойной клик по ячейке — открытие файла на нужной строке."
> "Никакой магии 'за кулисами'. Только честное, видимое соответствие."

**В спеках**: Двунаправленная запись через `viewApi.updateRecord()` задокументирована в `V2_SPEC §6.3`. Но **UX-слой прозрачности** — визуальное соответствие поле↔строка-файла — нигде не описан. Это отдельная фича с нетривиальной реализацией (нужен активный split-view между table и note preview с highlight).

**Последствие**: Пользователь не понимает, что именно меняется в файле. Связь "ячейка ↔ frontmatter поле" остаётся магией.

**Требуемое действие**: Тикет `#060 — Field transparency: column hover → file line highlight`. Архитектурно: при hover на column header показывать tooltip с именем frontmatter-ключа. При double-click на cell → `app.workspace.openLinkText(recordId, '', false)` с перемоткой на нужную строку через Obsidian API.

### 1.4 «Шаблоны как стартовая точка» — система, не один демо
**VISION §7 (дословно):**
> "'Я веду клиентов' — готовый набор баз, представлений и связей."
> "'Я тренируюсь' — другой набор."
> "'Я веду дневник проекта' — третий."
> "Можно начать работать за пять минут, а не за пять часов."

**В спеках**: `#046` (done) = один B2B демо-проект. Это один шаблон из трёх названных в Vision. Онбординг-флоу «выбери свой профиль → получи готовый набор» нигде не описан как система.

**Требуемое действие**: Тикет `#061 — Template Library: 3 starter profiles (clients / fitness / journal)`. На уровне `CreateProject.svelte` — выбор профиля при создании, каждый профиль генерирует свой набор баз и дашборда.

## Раздел 2: Конфликты «Vision говорит X, спек делает Y»

### 2.1 Drag-to-link vs «Link to...» context menu

**VISION §7:**
> "Drag & drop как основной способ выражения связей. Хочу связать клиента с сеансом — **тащу карточку на карточку**. Хочу добавить колонку — **тащу поле из бокового списка в заголовок таблицы**."

**V2_SPEC §5.3:**
> "В UI — через **контекстное меню** блока 'Link to...' → выбор блока-мастера → выбор поля связи."

**UI_DESIGN_ARCHITECTURE.md §4.3** (Design Gaps):
> "Column drag reorder — Not supported — P2"
(Drag для колонок = P2 gap, drag для выражения связей — не упомянут вообще)

**Это прямой конфликт.** Спек выбрал более реализуемый подход (контекстное меню), но VISION явно называет drag-to-link «основным способом». Ни один документ не описывает этот компромисс.

**Требуемое действие**: Зафиксировать осознанное решение: «Drag-to-link — это V3 vision. В V2 — context menu как stepping stone. Причина: drag-to-link требует сложного DnD между разными database-call блоками, что блокирует весь V2.» Добавить в BACKLOG как `#062 — Drag-to-link: drag card onto block to set relation (V3)`.

### 2.2 Timeline в архиве, но в Vision — центральный элемент

**VISION §1:**
> "Зоны представлений: …**временная шкала**…"
> (описывается в первой сцене как часть нормального рабочего экрана)

**V2_SPEC §4:**
> "timeline → ❌ Удалить (P3, зависит от calendar engine) → в archive/dashboard-v1"

**Проблема не в самом решении** — архивация Timeline технически обоснована. Проблема в том, что нигде нет фразы: "Мы знаем, что Vision упоминает Timeline как часть рабочей поверхности. Вот почему мы деферируем его в V2 и что нужно, чтобы вернуть в V3." Компромисс невидим.

**Требуемое действие**: В `DASHBOARD_V2_SPEC.md §4` к строке Timeline добавить footnote с обоснованием трейдоффа. В BACKLOG добавить `#063 — Timeline view: V3 roadmap item` со статусом DEFERRED + причиной.

### 2.3 Inline cell editing: gap P0 или designed feature?

**UI_DESIGN_ARCHITECTURE.md §3.4** (ASCII DataTable schema) показывает:
```
│  click → create record inline, focus first cell
```
И в §3.6 (Record Expand Panel):
```
│  Property click → inline edit (input or select dropdown)
```

Но §6 (Design Gaps) одновременно говорит:
```
| Click cell → inline edit | Click → SlideInPanel | 🔴 requires modal | P0 |
```

**Это внутреннее противоречие**: в одном разделе inline edit присутствует в дизайне, в другом — является gap. Документ сам с собой не согласен.

**Требуемое действие**: Определиться: inline edit — это часть дизайна #051 (тогда убрать из gaps) или отдельный тикет (тогда убрать из ASCII-схем). Рекомендация: включить в скоуп #051 явно.

## Раздел 3: Технические некорректности

### 3.1 Стейл baseline в DASHBOARD_V2_SPEC.md §11

```
ФАЙЛ: docs/internal/DASHBOARD_V2_SPEC.md строка 680
ТЕКСТ: npm test  # baseline ≥ 139 suites / 2099 tests
ПРАВИЛЬНО: npm test  # baseline ≥ 134 suites / 2020 tests
```

Этот документ — источник правды для разработчиков. Стейл число в гейте CI — прямой риск: тестер может посчитать 134/2020 за регрессию.

### 3.2 Хардкод hsl() в UI_DESIGN_ARCHITECTURE.md

```
ФАЙЛ: docs/internal/UI_DESIGN_ARCHITECTURE.md §2.4
ТЕКСТ:
  state.success → hsl(122, 40%, 45%)
  state.warning → hsl(38, 90%, 50%)
```

Это в документации, но документация формирует ожидания имплементации. Правильный паттерн — `--ppp-color-success: var(--color-green)` (Obsidian CSS variable). Жёсткие hsl() не адаптируются под тему пользователя.

**CLAUDE.md инвариант**: "No hardcoded hex colors in src/. Use design tokens or palette store."

### 3.3 Stale acceptance в #047 BACKLOG

```
ФАЙЛ: docs/internal/BACKLOG.md строка 498
ТЕКСТ: Gates: tsc 0 errors ✅ / 139 suites / 2099 tests PASS ✅
```

Это историческая запись (baseline в момент #047 действительно был таким). Но разработчик читающий BACKLOG может запутаться. Рекомендуется добавить пометку `(baseline at time of PR)`.

## Раздел 5: Пропущенные UX-паттерны (не в Vision, но обязательны)

Три паттерна, которые Notion имеет и которые Vision подразумевает, но явно не называет:

### 5.1 Empty states для Canvas (нет виджетов)

В `UI_DESIGN_ARCHITECTURE.md §3.10` определены empty states для таблицы и чарта, но не для самого канваса. Если пользователь создал Dashboard и ещё не добавил ни одного блока — что он видит?

```
Нужно:
┌──────────────────────────────────────────────────────────┐
│                                                          │
│         Ваш дашборд пока пуст                           │
│                                                          │
│   [+ Добавить блок данных]  [Выбрать шаблон]             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 5.2 «Добавить поле» — field creation flow

VISION §2: "Добавляю колонку — плагин предлагает: 'Создать поле во всех заметках?'"

Нигде в спеках не описан UX этого флоу. Как выглядит confirmation dialog? Что происходит с существующими записями? Это касается `viewApi.addField()` или frontmatter bulk-write — нет спека.

### 5.3 Contextual tooltips с человеческим языком

VISION §7: "Не 'включить агрегацию по реляционному ключу', а 'посчитать сумму по группе — например, выручку по каждому клиенту'"

Это требование к **тону и языку** всех tooltip'ов в интерфейсе. Нет ни одного docs/internal документа, задающего этот tone of voice.

## Раздел 7: Ключевой вывод — что изменить в подходе

Документация делает отличную работу для **инженерной команды** (механизм, API, контракты). Она делает слабую работу для **продуктовой** стороны — пользовательские ощущения, которые Vision описывает.

**Конкретная проблема**: Vision написан от первого лица, от сцены к сцене. Спеки написаны от компонента к компоненту. Между ними нет «сшивки» — нет документа, который берёт каждую сцену Vision и говорит «вот как она реализована в коде».

**Рекомендация**: Для каждой из 8 сцен Vision создать строку в tracking-таблице:

| Сцена Vision | Ключевые требования | Тикеты | Статус |
|---|---|---|---|
| Сцена 1: открываю дашборд | Stats-виджеты, multi-view canvas | #044, #052, #053, #065 | Partial |
| Сцена 2: база из заметок | folder=database, add field flow | — | 🔴 no UI spec |
| Сцена 3: двунаправленность | inline edit, transparency | #051, #060 | Partial |
| Сцена 4: связи-как-нервная-система | Relation, Rollup, SubBase | #009, #010, #044 | ✅ done |
| Сцена 5: фильтры как базы | native-query, filter→database | #045.2, #048 | Partial |
| Сцена 6: проактивные подсказки | SmartSuggest engine | #059 | 🔴 не начато |
| Сцена 7: интуитивность | drag-to-link, templates, tooltips | #061, #062, #065 | 🔴 не начато |
| Сцена 8: дашборд=markdown | YAML config format | #066 | 🔴 не начато |

Это должно стать живым документом, обновляемым после каждого milestone — чтобы всегда было видно, где продукт приближается к Vision, а где отстаёт.
