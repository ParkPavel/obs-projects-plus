# UT2026-D — Config Provenance: генераторы, миграции и реактивность конфигов

> Домен: #072 (stats «—»), #071 (cover config не применяется), L3 из UT2026-A.
> Статус: СОГЛАСОВАНО; #072 реализуется немедленно, #071 — после репродукции.

## Природа проблемы

**Кейс #072**: R5-004 переименовал агрегацию `"count"` → `"count_total"` и добавил миграцию
`migrateAggregationCount` для сохранённых конфигов. Но `demoProject.ts:260-261` — генератор —
продолжает порождать `"count"`. Миграция и генератор разъехались, потому что миграция чинит
прошлое, а **никто не зафиксировал правило для будущего**: генераторы обязаны эмитить
актуальную схему.

**Кейс #071**: панель конфига диспатчит change, проводка до saveConfig корректна, но select
визуально не меняется — класс проблем «конфиг ушёл вниз, но не вернулся в props» (или вернулся
с потерей реактивности). Тот же разрыв контракта, с другой стороны: один путь данных,
два конца которого никто не проверяет вместе.

## Контракт

**P1. Генератор эмитит только актуальную схему.** Любой код, порождающий конфиги
(demoProject, widgetTemplates, quick-actions, миграционные дефолты), производит значения,
которые проходят все миграции как no-op.
Enforcement: jest `configProvenance.test.ts` — для каждого генератора:
`migrate(generate()) === generate()` (deep-equal, миграции идемпотентны на свежем выводе).

**P2. Конфиг — круговой контракт.** Панель конфига обязана отражать сохранённое состояние из
props, а не локальную копию: после `dispatch("change")` единственный источник истины —
`widget.config`, прилетающий сверху. Локальные снапшоты (`$: cfg = config`) допустимы только
как derived, без шансов пережить prop-обновление.
Enforcement: компонент-тест на панель: change → новый props → DOM селекта отражает новое значение.

**P3. Переименование значения схемы = тройка.** Новое значение + миграция + грep по генераторам
в одном коммите. Чек добавлен в CLAUDE.md (см. финальную правку).

## Заведомо решаемые связанные проблемы

- Любое будущее переименование агрегаций/операторов фильтра не сможет тихо сломать демо
  и шаблоны — тест P1 упадёт.
- Первый экран нового пользователя (демо) гарантированно валиден относительно текущей схемы.
- Класс «настройка не применилась» получает воспроизводимый тест-паттерн (P2) вместо
  одноразовых фиксов.
