# UT2026-C — Color Contract: единый контракт цвета записи

> Домен: #070. Статус: КОНЦЕПТ для analysis-сессии (L) — кодировать после согласования.

## Природа проблемы

Цвет записи реализован трижды, тремя независимыми командами-сессиями:

| # | Механизм | Кто пишет | Кто читает |
|---|---|---|---|
| 1 | Цветовые правила проекта (`getRecordColor`, colorFilter conditions) | настройки проекта | все вью через prop |
| 2 | `config.eventColorField` — hex в frontmatter записи | popup календаря (`CalendarView.svelte:1094`) | календарь (`:1166,1977,2058`) |
| 3 | Эвристика имени поля `COLOR_FIELDS` | EditNote/FieldControl рендерят color-picker | никто системно |

Дефект пользователя («в попапе работает, в параметрах заметки — нет») — прямое следствие:
запись из EditNote идёт тем же `api.updateRecord`, но (а) приоритет 1 vs 2 не специфицирован,
(б) эвристика 3 продублирована в двух файлах и не связана с `eventColorField` конфига,
(в) календарь кэширует обработанные события (`lastProcessedVersion`) и инвалидация после
внешнего (не-календарного) апдейта не гарантирована.

## Контракт (целевой)

**C1. Резолюция цвета — одна функция.** `resolveRecordColor(record, project, viewConfig): string | null`
в `src/lib/colors/` с зафиксированным приоритетом:
явный per-record цвет (`eventColorField`) → правило проекта (colorFilter) → null (тема).
Обоснование: explicit beats rule — пользовательский жест сильнее декларативного правила
(совпадает с текущим ощущением «попап работает правильно»).

**C2. Детекция color-поля — одна точка.** Источник правды — `viewConfig.eventColorField` /
field typeHint, а НЕ substring имени. Эвристика COLOR_FIELDS остаётся только как fallback
для рендера пикера и живёт в ОДНОМ модуле (`src/lib/colors/detect.ts`), импортируемом
EditNote и FieldControl.

**C3. Инвалидация — событие, не совпадение.** Внешний `updateRecord` обязан инвалидировать
производные кэши вью (календарь: сброс `lastProcessedVersion` подпиской на frame-версию,
не только из собственных хендлеров).

**C4. Без миграции данных.** Hex в frontmatter остаётся как есть; меняется только код резолюции.

## План analysis-сессии (#070)

1. Семантический анализ всех читателей цвета (calendar, board, gallery, table, database-call).
2. Подтвердить точный сценарий рассинхрона EditNote → календарь (консоль + лог calendarLogger).
3. Спроектировать `resolveRecordColor` + `detect.ts`, мигрировать читателей по одному.
4. Jest: таблица приоритетов C1 (explicit/rule/none × все вью-конфиги).

## Заведомо решаемые связанные проблемы

- Board/Gallery, получающие цвет иным путём, выравниваются автоматически (C1 один для всех).
- Будущий «цвет по формуле» (Vision) встраивается в приоритетную цепочку, а не четвёртым механизмом.
- Дубль эвристики (EditNote vs FieldControl) устранён структурно (C2).
