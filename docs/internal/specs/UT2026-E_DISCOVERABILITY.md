# UT2026-E — Discoverability: обнаруживаемость возможностей (Vision §7)

> Домен: #075 (конвейер), #076 (путь баз/суб-баз), #077 (фильтры + мат-движок).
> Статус: КОНЦЕПТ для senior-designer; кодировать после дизайн-спеки.

## Природа проблемы

Тестирование показало повторяющийся паттерн: **функциональность существует, путь к ней — нет**.
Конвейер трансформаций, Матрёшка суб-баз, FilterPanel, FormulaConstructor — всё реализовано
и покрыто тестами, но пользователь не может ответить «что это и как этим пользоваться».
Это не дыры функций — это дыры **повествования интерфейса**. Точечные подсказки уже делались
(#047, #065, #059) и работают; не хватает системного слоя.

## Принципы (из Vision §7, обязательны для всех трёх тикетов)

**D1. Объяснение на месте, на языке задач.** Не «Unnest», а «развернуть список в строки —
по одному участнику на строку». Каждый управляющий элемент — tooltip с примером.

**D2. Прогрессивное раскрытие.** Продвинутое появляется, когда пользователь в него упирается:
конвейер предлагается из контекста данных (массивы → Unnest-подсказка УЖЕ есть в
PipelineEditor — паттерн правильный, его надо вынести на уровень входа), суб-база — из
Relation-поля (стыкуется со SmartSuggest #059: подсказка и есть канал discovery).

**D3. Один глагол — одна кнопка.** «fx» без подписи не считается входной точкой.
Каждая способность имеет именованный вход в тулбаре или контекстном меню.

## Карта входных точек (целевая, для дизайн-спеки)

|---|---|---|

| Суб-базы / Матрёшка (#076) | нет UI-входа | CTA из Relation-поля (через SmartSuggestionBus), пункт «+ суб-база» в database-call, готовая суб-база в демо |
