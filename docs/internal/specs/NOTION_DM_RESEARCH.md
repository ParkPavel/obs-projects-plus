## 1. Как это устроено в Notion (модель, не скриншоты)

**Ключевой факт: в Notion НЕТ пользовательского «конвейера трансформаций».**
Вся работа с данными разложена по местам, каждое — мгновенное:

| Операция | Где живёт в Notion | Как применяется |
|---|---|---|
| Фильтр | pills под тулбаром вью + «View options ▸ Filter»; quick filter по клику на свойство | **мгновенно** при изменении условия, без Save; AND/OR-группы вложенностью до 3 |
| Сортировка | «View options ▸ Sort» — упорядоченный список правил (drag) | мгновенно |
| Группировка | «View options ▸ Group» (+ sub-groups вторым уровнем) | мгновенно; collapse запоминается |
| Агрегация | футер таблицы per-column «Calculate», заголовки групп на доске | мгновенно |
| Вычисление | Formula-СВОЙСТВО базы (редактор формул с автокомплитом и live-результатом) | результат пересчитывается живьём при наборе |
| Cross-DB | Relation-свойство + Rollup-свойство (источник: relation → поле → функция) | конфиг свойством, не пайплайном |
| Развёртка | нет аналога unnest; массивы = multi-select/relation, группировка раскрывает их сама | — |
| Сводные/чарты | Chart view: тип + Ось X (what to group by) + Ось Y (count/sum свойства) | конфиг — попап при создании вью, мгновенный re-render |
| «Скрыто всё» | empty-state вида всегда показывает активные фильтры pill'ами — видно ЧТО скрыло | 1 клик до снятия |

**Принципы, которые делают это «просто»:**
1. **Нет режима редактирования данных-вида** — нет Save/Cancel; любое изменение применяется
   сразу, страховка — Undo (Ctrl+Z) и видимость правил (pills).
2. **Операция живёт у объекта**: про колонку — в колонке, про вид — в «View options», про
   связь — в свойстве. Никакого общего «процессора».
3. **Каждое правило видно и снимается одним кликом** (pill с ×).
4. **Live-результат всегда на экране** — данные за попапом меняются, пока крутишь условие.

## 2. Перевод на наш плагин: РАСЩЕПЛЕНИЕ КОНВЕЙЕРА (эпик #099)

Наш конвейер — это власть 8 операций в одном модальном редакторе с Save. Notion-модель
требует разнести его по местам грамматики; сам конвейер остаётся только для того, чему
в Notion нет аналога (наше преимущество — unnest/pivot/join), и становится живым.

| Шаг конвейера | Куда переезжает | Статус механики |
|---|---|---|
| filter | **Filter-pills в ControlBar вида** (builder-попап = FilterPanel), мгновенно | FilterPanel есть; вынести вход + live |
| group-by | уже в меню колонки (W1) + «View options» | ✅ сделано W1 |
| aggregate | уже Calculate в меню колонки/футере; для чартов — конфиг чарта | ✅/частично |
| compute | Formula-ПОЛЕ через FormulaConstructor (#077) — свойство, не шаг | W3 |
| join | свойство БЛОКА: «Дополнительный источник» в DatabaseCallSettings | W3 (#091-смежно) |
| unnest | свойство БЛОКА: «Развернуть список: <поле>» — один переключатель + SmartSuggest-подсказка уже есть | #099 |
| pivot/unpivot | остаются в «Расширенных преобразованиях» (power-режим) | #099 |

**Новый конвейер (то, что остаётся): «Расширенные преобразования»**
- вход: из меню виджета «⋯», помечен как продвинутый;
- **без Save/Отмена**: каждый чек применяется немедленно к виджету за попапом
  (executeTransform уже возвращает inputRowCount/outputRowCount — счётчики готовы);
- каждый шаг показывает **N → M записей** живьём; шаг, давший 0, подсвечен предупреждением
  с кнопкой «Отключить шаг» (замещает «Нет данных» без выхода — #092);
- поле выбирается типизированным пикером (иконки типов, без housekeeping-полей — R4-скрин);
- operator/value — те же контролы, что в FilterPanel (один builder на продукт).

## 3. Аудит реактивности (по модулям, факты из кода)

| Контур | Состояние | Вердикт |
|---|---|---|
| vault modify/create/delete/rename → externalFrameCache.clear + bump (App.svelte:98) | есть | ✅ |
| dataFrame store mutators → transformCache.invalidateAll (синхронно) | есть | ✅ |
| Таблица/виджеты ← transformedFrame | реактивно по props | ✅ |
| **PipelineEditor** | draft применяется ТОЛЬКО по «Сохранить»; за попапом «Нет данных» живёт старая версия | 🔴 ядро жалобы — лечится №2 |
| **Конфиг-панели** (cover #071 и др.) | apply-on-change есть, но select-ребиндинг под вопросом; нет optimistic-эха в части панелей | ⚠ W2 #090/#071 |
| **EditNote** | valuesSnapshot — снапшот на открытие; внешние изменения записи не подтягиваются в открытую модалку | ⚠ тикет #101 |
| **CalendarView** | свой records-стейт + lastProcessedVersion + локальные «optimistic» копии цветов | ⚠ перепроектировать в #078 (декомпозиция) |
| Selection Bus | идемпотентный store, мгновенный | ✅ |
| Right-frames preload | по referencedIds + bump | ✅ |

Вывод: фундамент реактивен; «страдает реактивность» = три точки (конвейер-Save,
панельный round-trip, модалка-снапшот) + календарь архитектурно.
