# UT2026-A — Legacy Containment: контракт изоляции архивного кода

> Домен: #068 (краш fields/groups), #073 (палитра), #074 (Table rebuild), demo data-table.
> Статус: СОГЛАСОВАНО КОНЦЕПТУАЛЬНО (пользовательский мандат 2026-06-11). #074 требует architect-план.

## Природа проблемы (почему точечный фикс — ошибка)

#056 «заархивировал» V1-виджеты переносом в `src/archive/dashboard-v1/` — но это был перенос
файлов, а не вывод из эксплуатации. Архив остался живым по трём каналам:

1. **Маршрутизация**: `WidgetHost.svelte:14` импортирует `DataTableWidget` из архива → архивный
   код исполняется в каждом дашборде с `data-table`.
2. **Создание**: `DashboardBlockPalette` рендерит весь `WIDGET_REGISTRY`, включая архивные типы —
   пользователь продолжает создавать новые экземпляры мёртвого кода.
3. **Генерация**: `demoProject.ts` сам создаёт `data-table` («Клиенты») — первый опыт нового
   пользователя проходит через архив.

Краш #068 — не баг таблицы, а **следствие отсутствия контракта**: код, который никто не
сопровождает (архив исключён из рефакторингов, token-свипов, emoji-свипов #047), неизбежно
деградирует, оставаясь в проде.

## Контракт (инвариант после внедрения)

**L1. Архив не исполняется.** Ничто в `src/` вне `src/archive/` не импортирует из `src/archive/`.
Enforcement: jest-инвариант `R0_4_archiveContainment.test.ts` — grep import-путей.

**L2. Архивные типы не создаются.** `WidgetMeta.legacy: true` для типов из fate-table V2 §4;
палитра показывает legacy-тип только если экземпляр уже есть на канвасе (поддержка старых
конфигов — да; новые экземпляры — нет).

**L3. Генераторы не порождают legacy.** demoProject, templates, quick-actions создают только
активные V2-типы. Enforcement: тест на демо-конфиг (см. UT2026-D).

**L4. Существующие пользовательские конфиги не ломаются.** Рендер legacy-виджета сохраняется
до миграции (#074): deprecation ≠ удаление. Путь вывода: data-table → `database-call` Table tab
(автомиграция конфига), остальные — по fate-table.
