## 2. Визуальный аудит интерфейса

> Методология: 7 слоёв визуального аудита по стандартам Notion UI.
> Слои: Навигация → Цвет → Типографика → Состояния компонентов → Charts → Calendar → Gallery.

### 2.1 Навигационная архитектура

**Структура:** 3 уровня навигации
```
Уровень 1: Табы проектов (База: Обзор | База: Kanban+ | Доска | Календарь | ...)
Уровень 2: View sub-tabs (Виджеты | Сетка | Схема | fx | + Добавить виджет)
Уровень 3: Preset tabs (Обзорный пресет | Конструктор формул)
```

| Элемент | Состояние | Проблема |
|---|---|---|
| Активный таб проекта | Чуть темнее фон | ⚠️ Слабый contrast active/inactive |
| View sub-tabs | Виджеты = фиол. pill | ⚠️ Остальные слабо различимы |
| Preset tabs (3-й уровень) | Только нижний бордер | ❌ Нет визуального веса |
| 3 уровня навигации | Существуют | ❌ Когнитивная перегрузка |

### 2.2 Dashboard View — детальный разбор

**Finance KPI виджет:**
- `67 Records | 35 Open | 33.6% Avg Progress` — число и лейбл в одну строку одним весом
- ❌ Нет иерархии label/value. Notion: крупное число + маленький лейбл под ним

**Summary виджет:**
- Метаданные: `67 records · pipeline steps: 0 · columns: 3` — технический текст, не user-friendly
- Метрики: `Сумма·estimate = 298.25 | Среднее·progress = 33.6% | Количество·name = 67`

**By Category (chart виджет) — КРИТИЧЕСКАЯ ПРОБЛЕМА:**
```
Ось X (27 категорий):
Invoразработкубухбазайностьладсэтиldaporрравленниразвийемаркетинг
```
❌ Все подписи рендерятся в одну точку → полная нечитаемость при 15+ категориях.
Notion: при переполнении — ротация 45°, обрезка, или limit N подписей.

**Journal (таблица в виджете):**
- Данные есть (67 записей), таблица работает
- Tooltip "Таблица данных" в правом нижнем углу — артефакт, лишний элемент

### 2.3 Board View (Kanban)

Колонки: `inbox | scheduled | todo | doing | + Добавить колонку`

**Заголовок колонки:** `{название}  ><  📌  📋  ⋮`
- ❌ Нет счётчика карточек рядом с названием (Notion: `inbox **3**` в сером)
- ❌ Нет цветовой маркировки колонок (Notion: цветной заголовок по умолчанию)

**Анатомия карточки — КРИТИЧЕСКАЯ ПРОБЛЕМА:**
```
priority          ← label gray ~11px
medium            ← value white ~14px

priority          ← ДУБЛИКАТ того же поля!
medium

tags
  идея  ui  тема  ← chips

type
идея
```
- ❌ **Название заметки (title) не показано** — в Notion title всегда первый и крупнейший элемент
- ❌ **Дублирование свойства `priority`** в одной карточке
- ⚠️ Все тэг-чипы одного фиолетового цвета (в Gallery те же тэги имеют разные цвета)

**Левый акцент карточки:** жёлтый/янтарный бордер — стабильный, работает.

### 4.3 Критический пропуск — незамкнутый реактивный контур

**Сейчас система работает в одну сторону:**
```
Obsidian файлы → Datasource → DataFrame → Formula/Transform → Дашборд
      ↑                                                            |
      └──────────────────── НЕТ ОБРАТНОЙ СВЯЗИ ←──────────────────┘
```

**Нужно:**
```
Obsidian файлы → Datasource → DataFrame → Formula/Transform → Дашборд
      ↑                 ↑                                          |
      │         vault.on("modify")                                  |
      └──── invalidate cache ←──── recalculate ←──────────────────┘
```

Кэш существует (`transformCache.ts`), инвалидация есть (`invalidateTransformCache()`), но **подписки на Obsidian vault events нет**. Пользователь не видит обновление при изменении файла.

### 4.4 Оценка объёма работ по автоматизации вычислений

| Блок | Задача | Трудоёмкость | Приоритет |
|---|---|---|---|
| **Блок 1** | Live recalculation при изменении файлов | 3–5 дн | 🔴 P0 |
| **Блок 1** | Visual indicator пересчёта | 1 дн | 🔴 P0 |
| **Блок 2** | Writeable formula fields | 5–7 дн | 🟠 P1 |
| **Блок 2** | Cycle detection в формулах | 2–3 дн | 🟠 P1 |
| **Блок 2** | Conflict resolution (ручное vs вычисленное) | 2 дн | 🟠 P1 |
| **Блок 3** | UI конструктор JOIN пайплайна | 7–10 дн | 🟡 P2 |
| **Блок 3** | Preview вычислений в реальном времени | 3 дн | 🟡 P2 |

**Итого: ~35–45 рабочих дней** (7–9 недель при одном разработчике)

**Ключевой инсайт:** 80% пользовательской ценности от "автоматизации вычислений" даёт только Блок 1 (3–5 дней). Всё остальное — углубление.

### 5.2 Пробелы в понимании

**Пробел 1 — Продуктовый vs технический приоритет**

Агент интерпретировал продуктовое видение как технический рефакторинг. Это было необходимо и сделано добросовестно. Но сейчас бэклог — это список технических задач, а не карта пути к живому продукту. Нет ответа на вопрос: *как пользователь за 3 шага создаёт живую базу со связями?*

В бэклоге V5.5–V5.6 есть суббазы и связи (структура). Но нет ни одного тикета на живое обновление при изменении файла. Это критический пропуск для цели "автоматизация вычислений через БД". → **Добавлен R5-016 (P0).**

**Пробел 3 — "Связи не только по строкам, но и по свойствам"**

Пользователь описал связь между **свойствами** разных заметок, а не только между записями целиком. R5-010 описывает bidirectional relations между суббазами, но это всё равно связь записей. Концепт связи свойств не выделен архитектурно — зафиксирован как known gap в SESSION_REPROMPT §7.

**Пробел 4 — Дизайн задокументирован, но не операционализирован**

DESIGN_CONCEPT_NOTION_AESTHETIC.md — хороший документ с правильными принципами. Но в REFACTOR_BACKLOG нет тикетов на конкретные компоненты: Board card без title, chart overflow, несогласованные тэги. → **Добавлены R5-017…R5-021.**

**Пробел 5 — Сессия 3 остановилась перед главным**

V5.8 (Dataview Adaptive Bridge) помечен DRAFT с пометкой "конкретные файлы определяются в сессии 3". Агент остановился перед самым важным шагом.

## 6. Стратегическое видение текущего этапа

### 6.1 Где находится проект

Проект находится **на переломе** между двумя режимами:
- Завершён режим: **"починить что сломано"** (V5.1–V5.7 — рефакторинг, очистка, унификация)
- Начинается режим: **"построить что должно жить"** (V5.8+ — реактивность, UX, продукт)

### 6.2 Главное противоречие

Проект пытается быть двумя продуктами одновременно:

**Продукт А** — удобный инструмент управления проектами для пользователей Obsidian (Notion-like experience, локально и офлайн)

**Продукт Б** — аналитическая платформа с transform pipeline, финансовыми формулами, scatter-графиками с R², cross-DB join

Оба продукта незавершены. Продукт А — нет двусторонних связей, нет живого пересчёта. Продукт Б — нет UI для настройки пайплайна без YAML-конфига.

### 6.3 Ключевая метафора

> Проект сейчас — **Ferrari с кузовом от Жигулей**.
> Двигатель (formula engine, transform pipeline, sub-bases) мощный и готов.
> Но первое впечатление не соответствует тому что внутри.

### 6.4 Что такое "автоматизация вычислений через БД" в реальности

Сейчас этого нет. Дашборд — это **отчёт**, а не **контрольная панель**. Разница принципиальная.
